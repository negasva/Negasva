Autenticación admin: rotar contraseña no invalida sesiones

P0 · Crítico

La clave HMAC es ADMIN_SESSION_SECRET || ADMIN_PASSWORD. Si existe ADMIN_SESSION_SECRET, rotar la contraseña no mata sesiones activas. Además, el IP de rate limit es spoofable vía XFF y las respuestas admin no tienen Cache-Control: no-store.

Prompt para Claude Code

/ponytail caveman mode on.

Lee audits/seguridad/autenticacion-admin/BUENAS_PRACTICAS.md antes de tocar código.

Fix A1 — rotar ADMIN_PASSWORD debe invalidar sesiones:
- lib/admin/session.ts:14 — el token HMAC incluye un hash de ADMIN_PASSWORD en el payload.
  Forma mínima: al firmar, añade un campo passwordHash: sha256(ADMIN_PASSWORD).slice(0,8).
  Al verificar, rechaza si no coincide con el hash actual.

Fix M2 — XFF spoofable en rate limit del login:
- lib/security/apiHelpers.ts:12-16 — en Vercel, preferir en orden:
  x-vercel-forwarded-for → x-real-ip → primer valor de x-forwarded-for.

Fix B1 — Cache-Control en respuestas admin:
- lib/security/apiHelpers.ts — en errorResponse y en un nuevo helper successAdminResponse,
  añadir el header Cache-Control: no-store, no-cache.
- app/api/admin/**/route.ts — usa el helper en todas las respuestas JSON de admin.

Commit: fix(admin): invalidar sesión al rotar password, IP real en rate limit, no-store en admin

RLS Supabase: is_admin() usa metadata editable por el usuario

P0 · Crítico

is_admin() lee user_metadata.role, que cualquier usuario autenticado puede autoeditar. Si el signup de Supabase está abierto, cualquiera puede darse rol admin. Además, /api/track lee orders con la anon key — incompatible con las políticas versionadas.

Prompt para Claude Code

/ponytail caveman mode on.

Lee audits/seguridad/rls-supabase/BUENAS_PRACTICAS.md antes de tocar código.

Fix A1 — is_admin() no debe confiar en user_metadata:
- supabase/migrations/ — crea nueva migración que reemplace:
    auth.jwt() -> 'user_metadata' ->> 'role' = 'admin'
  por:
    auth.jwt() -> 'app_metadata' ->> 'role' = 'admin'
  en TODOS los lugares donde aparezca (006_admin_tables.sql, 019_order_photos.sql y cualquier otro).
  app_metadata solo es modificable con service role key.

Fix A2 — /api/track debe usar service role, no anon key:
- app/api/track/route.ts — cambia getSupabase() (anon) por createServiceClient() (service role)
  para la consulta a orders. La lógica de matching (orderId + email) se queda en el servidor.
  Elimina cualquier política SELECT anon sobre la tabla orders si existe fuera del repo.

Fix M2 — INSERT anon en newsletter saltea rate limit:
- supabase/migrations/ — nueva migración: elimina la política WITH CHECK (true) de INSERT anon
  en newsletter_subscribers. El insert debe hacerse solo desde /api/newsletter con service role.

Commit: fix(rls): usar app_metadata en is_admin, service role en /api/track, quitar insert anon newsletter

Webhooks: ninguno coteja el monto pagado contra el pedido

P0 · Crítico

Los 4 webhooks marcan un pedido como pagado solo por coincidencia de referencia, sin comparar el importe capturado contra orders.amount_total. Un pago parcial o con importe manipulado marca el pedido como pagado. Además, Mercado Pago no verifica firma HMAC.

Prompt para Claude Code

/ponytail caveman mode on.

Lee audits/pagos-y-dinero/webhooks/BUENAS_PRACTICAS.md antes de tocar código.

Fix A1 — cotejar monto pagado contra orders.amount_total en todos los webhooks:
Crea helper lib/payments/verifyAmount.ts:
  verifyAmount(paid: number, currency: string, order: {amount_total: number, currency: string}): boolean
  Retorna true solo si moneda coincide y paid >= order.amount_total * 0.99 (1% de margen por redondeo).

Aplica en cada handler justo antes de marcar paid:
- app/api/webhooks/paypal/route.ts — usa amount.value del evento CAPTURE.COMPLETED.
- app/api/webhooks/mercadopago/route.ts — usa transaction_amount del pago re-fetched.
- app/api/webhooks/stripe/route.ts — usa amount_total del PaymentIntent.
- app/api/webhooks/wompi/route.ts — usa amount_in_cents / 100.
Si verifyAmount falla: responde 200 (para que el provider no reintente) pero NO marques paid.
Loguea con console.error('amount mismatch', {paid, expected, ref}).

Fix M1 — Mercado Pago sin firma HMAC:
- app/api/webhooks/mercadopago/route.ts — verifica x-signature + x-request-id con HMAC-SHA256
  usando process.env.MERCADOPAGO_WEBHOOK_SECRET antes del re-fetch.
  Rechaza con 401 si la firma no coincide.

Fix M3 — post-proceso de cupón/email no aislado del update de estado:
- En cada webhook, separa el bloque update de status (crítico) del bloque post-proceso
  (recordDiscountCodeUse + notifyNewOrder). Pon el post-proceso en un try/catch propio
  que loguee el error pero no lance; el update de estado ya está confirmado.

Commit: fix(webhooks): cotejar monto pagado, firma HMAC en MP, aislar post-proceso
